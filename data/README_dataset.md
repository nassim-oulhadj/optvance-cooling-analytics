# README_dataset.md, Doha DC-1 Cooling Telemetry (Synthetic)

## Overview
`cooling_telemetry_doha_dc1.csv` is a synthetic 12-month, 15-minute-interval
dataset (35,040 rows) representing cooling plant telemetry from the Doha
DC-1 reference campus (30 MW, Qatar), generated by `generate_dataset.py`
per the CoolingHealthSentinel pre-onboarding assignment spec (Section 1).

Run `python generate_dataset.py` to regenerate (seed = 42, fully
deterministic). The script prints a validation report against every spec
range and the anomaly schedule on each run.

## Generation Methodology
The script builds the dataset in three layers so anomaly effects are
physically consistent rather than hardcoded onto output columns:

1. **Root signals**, weather (diurnal + seasonal), IT load (baseline +
   AI bursts), and independent component-health parameters (chiller age
   decline, tower fouling drift, pump wear drift).
2. **Anomaly injection**, the 8 scheduled events mutate root signals
   only, within their time window.
3. **Derived signals**, all dependent columns (chiller temps, COP, PUE,
   Cooling Health Score) are computed once, after anomaly mutation, so
   each anomaly's effects propagate through the same formulas used for
   normal operation.

## Feature Definitions

| Feature | Unit | Range | Construction |
|---|---|---|---|
| `outdoor_temp_c` | °C | 18–48 | Seasonal cosine (peak ~mid-July, Qatar climate) + diurnal cosine (peak 15:00) + noise |
| `outdoor_humidity_pct` | % | 10–85 | Linear inverse function of `outdoor_temp_c` + noise |
| `it_load_mw` | MW | 8–28 | Baseline ~12 MW (daily cycle + noise); 2–5 AI workload bursts/day raise load to 24–28 MW for 30 min–4 h |
| `chiller_inlet_temp_c` | °C | 6–14 | 7°C target + 0.15×(it_load − 12) + noise |
| `chiller_outlet_temp_c` | °C | 12–22 | inlet + ~5°C + noise |
| `chiller_cop` |, | 2.8–6.5 | Base COP × age decline (8%/yr linear) × month-8 step factor × outdoor-temp efficiency factor + noise |
| `cooling_tower_approach_c` | °C | 2–8 | Baseline 3.5°C + slow fouling drift over the year + noise |
| `crah_delta_t_c` | °C | 8–18 | 10°C baseline + 0.2×(it_load − 12) + noise |
| `pump_vibration_mms` | mm/s | 0.5–4.5 | Baseline 1.0 + slow bearing-wear drift over the year + noise |
| `pump_flow_rate_ls` | L/s | 80–220 | Function of it_load and pump vibration (wear reduces flow) + noise |
| `water_conductivity_us` | µS/cm | 200–900 | Function of `cooling_tower_approach_c` (shared fouling driver) + slow drift + noise |
| `pue` |, | 1.05–1.45 | Linear function of `chiller_cop` (lower COP → higher PUE) + noise |
| `cooling_health_score` | 0–100 (range 55–98) | Nonlinear, noisy composite of 7 normalized health terms (see below) |
| `is_anomaly` | 0/1 |, | 1 if the row is part of either anomaly population (structured or background), see "Anomaly Design" below |
| `anomaly_type` | string |, | Name of the active structured anomaly, `"background_sensor_drift"` for background rows, `""` otherwise |
| `anomaly_class` | string |, | `"structured"`, `"background"`, or `"none"`, the two-population grouping variable; see below |

### Cooling Health Score (CHS)
CHS is a weighted composite of seven normalized 0–1 "health" terms,
COP (0.20), tower approach (0.13), pump vibration (0.13), pump flow
(0.13), water conductivity (0.13), PUE (0.14), and a thermal-stress term
combining chiller inlet temp and CRAH delta-T (0.14), plus a
compounding penalty term (extra deduction when the worst of the four
core terms is also poor, modeling correlated multi-system failure) and
Gaussian noise. The result is rescaled to the spec's 55–98 range.

This is deliberately **not** a clean linear function of any single
input feature: the compounding penalty introduces a multiplicative
nonlinearity, and the noise term prevents exact reconstruction. This
matters for Day 4 (GBR training), none of the 7 terms or `is_anomaly`
should be treated as leakage-safe to drop casually; check correlation
with CHS as part of EDA before deciding what to exclude from modeling.

## Anomaly Design, Two Populations

Section 1.2 of the assignment lists `is_anomaly` prevalence as "~3%"
(~1,051 rows/year). Section 1.3 specifies the construction rule as "8
anomaly events of 2–6 hours duration," which caps out at ~192 rows/year
(~0.55%), these two requirements cannot both be literally true (8 × 6h
× 4 steps/hour = 192 rows max, nowhere near 1,051).

This was raised with the instructor and resolved as follows: **keep
both, as two distinct anomaly populations**, rather than picking one
spec line over the other.

### 1. Structured anomalies (8 events, `anomaly_class = "structured"`)
The original 8 events from Section 1.3, multi-hour (2–6h), each
degrading ≥3 correlated sensor channels together, representing genuine
failure-precursor incidents (chiller surge, bearing wear onset, fouling
spike, etc.). See the anomaly table further down for the full list.
**123 rows** in the current generation (seed 42).

### 2. Background anomalies (`anomaly_class = "background"`)
A new layer simulating the low-grade sensor drift and marginal
exceedances that occur continuously in a real plant, which an
IsolationForest needs to learn to distinguish from genuine failure
precursors. Construction:

- ~2.7% of rows still flagged normal after the 8 structured events are
  sampled (943 rows in the current generation, from a pool of 34,917).
- For each sampled row, 3 randomly chosen telemetry columns (out of all
  12) receive independent Gaussian noise scaled to 1.5–2x that column's
  normal-population standard deviation, then are re-clipped to the
  column's spec range.
- **`cooling_health_score` is deliberately left untouched** for these
  rows. The premise is that this is a sensor-reading problem, not a real
  decline in system health, the chiller, pump, and tower are actually
  fine, only the measurement is noisy. This is the intended contrast
  with the structured population: background rows should look
  anomalous in raw sensor-feature space but should *not* show a
  corresponding CHS drop, while structured rows should show both.

Combined: **1,066 of 35,040 rows (3.042%)** are flagged `is_anomaly = 1`
Reconciling both spec lines rather than satisfying only one.

**Validation (confirms the design works as intended):**

| `anomaly_class` | rows | mean CHS | std CHS |
|---|---|---|---|
| none (normal) | 33,974 | 80.54 | 5.19 |
| background | 943 | 80.78 | 5.24 |
| structured | 123 | 75.46 | 4.08 |

Background-row mean CHS (80.78) is statistically indistinguishable from
normal (80.54), as designed, sensor noise alone doesn't move the health
score. Structured-row mean CHS (75.46) is clearly degraded, about 5
points below normal. The two populations behave as differently as
intended, which sets up Day 5's analysis of whether IsolationForest
(trained on raw/engineered sensor features, which can't see
`cooling_health_score` or the ground-truth label) can actually tell the
two apart, or whether it just flags both as equally anomalous because it
has no way to know one is a real failure and the other is sensor noise.

### Structured Anomaly Injection Logic
Eight anomaly types are injected once each, non-overlapping, spread
across the year, each lasting 2–6 hours and degrading at least 3 sensor
channels simultaneously:

| Anomaly | Duration | Channels affected |
|---|---|---|
| `cooling_tower_fouling` | 4–6 h | tower approach ↑, conductivity ↑ (coupled), COP ↓ (coupled) |
| `pump_bearing_wear` | 2–4 h | vibration ↑, flow ↓ (coupled), PUE ↑ (coupled) |
| `refrigerant_leak_step` | 4–6 h | COP ↓ sharply, PUE ↑ (coupled). **Scheduled in month 8 (August)** and leaves a permanent ~10–15% additional COP step-down for the rest of the year, on top of the existing 8%/year linear decline |
| `crah_filter_clog` | 3–5 h | CRAH delta-T ↑, chiller inlet temp ↑, COP ↓ (coupled) |
| `chiller_control_oscillation` | 2–4 h | inlet/outlet temp oscillate, COP ↓ (coupled) |
| `flow_sensor_fault` | 2–3 h | pump flow reading frozen at pre-fault value, tower approach ↑ (coupled), a data-quality fault co-occurring with a mild real stress event, not flagged via flow alone |
| `ai_thermal_overload` | 4–6 h | IT load pushed above normal burst ceiling, CRAH delta-T ↑, chiller inlet temp ↑, COP ↓ (coupled) |
| `tower_fan_degradation` | 3–5 h | tower approach ↑ sharply, COP ↓ (coupled), PUE ↑ (coupled) |

Validation: comparing each event's mean CHS to its own 12-hour
pre-window baseline (not the annual mean, which is confounded by
seasonality) shows a 1.6–6.0 point real drop for every event. See the
script's printed output for the exact figures on a given run.

## Spec Tension, Resolved, Not Silently Picked
See "Anomaly Design, Two Populations" above for the full reconciliation.
In short: Section 1.2's "~3% prevalence" and Section 1.3's "8 events of
2–6 hours" (≤192 rows, ~0.55%) were mathematically incompatible as
written. Per instructor guidance, both are honored by treating them as
two distinct anomaly populations (structured + background) rather than
satisfying one spec line at the expense of the other.
